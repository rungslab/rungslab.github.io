# Rungs 신규 엔트리 프로젝트 작업 규칙

앞으로 새 엔트리(Entry) 프로그램을 만들 때는 아래 순서를 그대로 따른다.

## 1. .ent 파일 직접 코딩

- `.ent` 파일은 gzip으로 압축된 tar 안에 `temp/project.json`이 들어있는 구조다 (unzip이 아니라 `tar xzf`로 연다).
- 새 블록 코드를 만들 때 사용하는 block `type` 값은 **반드시 검증된 것만 사용**한다:
  - `워크시트/Entry files/`에 있는 기존 실제 프로젝트 파일들을 파싱해서 실제 사용된 block type과 params 구조를 확인하거나,
  - entrylabs/entryjs 공식 소스(`src/playground/blocks/*.js`)에서 확인한다.
  - 둘 다 확인 안 된 블록(특히 마이크 음량, 얼굴/손 인식 등 AI 확장 블록처럼 흔치 않은 것)은 **절대 추측해서 넣지 않는다** — 잘못 넣으면 파일 전체가 깨질 수 있다.
  - 확인이 안 된 블록이 필요한 자리에는 검증된 블록으로 안전하게 대체(예: 마이크 감지 대신 스페이스바)해두고, 사용자가 엔트리 편집기에서 한 블록만 교체하면 되도록 정확한 스왑 방법을 안내한다.
- 빌드 후에는 항상 round-trip 검증을 한다: 생성한 `.ent`를 다시 풀어서 JSON 파싱 + 사용된 block type 목록이 전부 검증된 목록 안에 있는지 확인.
- gzip/tar 포맷은 `tarfile.USTAR_FORMAT` + `mtime=0`으로 만들어서 실제 엔트리가 export하는 파일과 최대한 비슷하게 맞춘다.
- 사용자가 엔트리에서 직접 수정한 뒤 다시 파일로 받으면, 그 파일이 이제부터의 정본(ground truth)이다 — 내가 만든 임시 버전은 대체하고, 실제 사용된 block type(예: `get_microphone_volume`)을 다음부터 검증된 목록에 추가한다.

## 2. 워크시트(docx + pdf) 제작

- 이 저장소에 `docx`(npm)는 설치가 안 되어 있고 npm 레지스트리 접근이 막혀 있다 → **python-docx**(`pip install python-docx --break-system-packages`)로 만든다.
- 기존 워크시트 스타일과 섹션 순서를 그대로 따른다(빠뜨리면 안 되는 전체 구성):
  1. 제목 배너(teal `0F7E9E`, 흰 글씨, 표 안에 셀 배경색으로 구현) + 부제목
  2. 인트로 문단 2개 + "이 워크시트는 실제로 완성되어 검증된 프로젝트 파일(...)의 내부 블록 구조를 그대로 분석하여 정리한 자료입니다" 방법론 문장
  3. 프로젝트 개요 (배너 + 표: 제작도구/사용기술/오브젝트 수/핵심개념/예상 소요시간)
  4. 학습 목표 (배너 + 불릿)
  5. 오브젝트 살펴보기 (배너 + 오브젝트 개수 문장 + 오브젝트별 이미지 카드[이미지+이름+설명]). **오브젝트 이미지도 실제 코스튬을 우선 사용**한다:
     - Claude in Chrome으로 프로젝트를 불러온 뒤, 오브젝트 선택 → "모양" 탭 → 왼쪽 코스튬 목록에서 원하는 코스튬 썸네일 **우클릭 → "이미지 파일로 내보내기"**를 누르면 실제 배경 투명 PNG/SVG가 사용자의 Downloads 폴더에 바로 저장된다(엔트리봇 같은 기본 제공 캐릭터도 이 방법으로 실제 원본을 받을 수 있다 — playentry.org 정적 경로로 직접 fetch하면 404가 남).
     - SVG로 받아지면 `convert -background none -density 300 in.svg out.png`(ImageMagick)로 투명 배경 PNG로 변환해서 쓴다.
     - 이 방법으로도 못 구하면(예: 프로젝트를 브라우저로 불러올 수 없는 상황) 커스텀 코스튬 PNG를 `.ent` tar 안에서 직접 추출하거나, 그것도 없으면 정직하게 라벨링한 간단한 아이콘을 그려서 대체한다 — 이미지 없이 텍스트만 쓰지 않는다.
  6. 순서대로 따라 만들기 (배너 + 번호 매긴 전체 제작 절차, 8단계 내외)
  7. 블록 구성 가이드 (배너 + 단계별 소제목 + 순서/블록/설명 표)
  8. 코드 살펴보기 (배너 + 설명 문장 + 블록 화면 이미지 + 블록 카테고리 태그 목록). **실제 캡처가 원칙**이다:
     - Claude in Chrome으로 playentry.org에 로그인한 뒤(비로그인 상태에서는 "오프라인 작품 불러오기" 메뉴 자체가 안 보임 — 반드시 로그인 필요), 상단 "+" 드롭다운 → "오프라인 작품 불러오기"로 실제 `.ent` 파일을 불러온다.
     - 각 오브젝트를 선택한 뒤 블록 조립소 빈 공간에서 **우클릭 → "모든 코드 이미지로 저장하기"**를 누르면 해당 오브젝트의 코드가 깔끔한 PNG로 사용자의 실제 Downloads 폴더에 바로 다운로드된다(파일명은 `{프로젝트id}_block.png`, 동일 프로젝트 내 여러 오브젝트를 연달아 저장하면 `(1)`, `(2)`로 자동 구분됨) — 이 방법이 화면을 스크린샷으로 캡처하는 것보다 훨씬 깔끔하고 확실하다.
     - 스크린샷 캡처 방식(`computer` 도구의 zoom+save_to_disk)은 대안으로만 쓴다. 단, 그 방식으로 저장된 파일은 Claude 세션의 별도 내부 캐시 경로에 생기며 `present_files`/직접 파일 접근이 막혀 있어 실질적으로 못 쓸 수 있다 — 반드시 "모든 코드 이미지로 저장하기"(Downloads 폴더 경유) 방식을 우선한다.
     - 이미지 삽입 후에는 설명 문장을 "실제 프로젝트 파일(.ent)을 엔트리 편집기에 직접 불러온 뒤, 편집기의 '모든 코드 이미지로 저장하기' 기능으로 내려받은 화면 그대로입니다"처럼 정직하게 실제 캡처임을 밝힌다.
     - 캡처한 화면에 보이는 실제 숫자값(임계값, 좌표 변화량, 대기 시간 등)이 워크시트 본문의 설명/표/체크리스트에 적어둔 숫자와 반드시 일치하는지 재확인한다 — 만들 때 썼던 임시값과 사용자가 실제로 바꾼 값이 다를 수 있다.
  9. 자주 발생하는 오류 (배너 + 불릿)
  10. 지금까지 무엇을 했나요? (배너 + 문단)
  11. 더 알아보기: [주제] (소제목, 배너 아님 + 문단 + 실생활 활용 예시 불릿 3개 내외)
  12. 아이디어 더하기 (소제목 + 확장 아이디어 불릿 3~4개)
  13. 페이지 나누기 후 "워크시트 — [프로젝트명]" 배너(dark navy) + 이름/날짜 빈칸 + 질문 + 체크리스트
- 오브젝트 살펴보기의 이미지 카드와 별도로 같은 내용을 텍스트 불릿으로 또 반복하지 않는다(중복).
- 브랜드 컬러: teal `0F7E9E`, yellow `FFC24B`, dark navy `1A1A2E`, 보조 accent `45C3DE/5FC75F/9F65E6/E06464/7C6FD6`.
- 주의할 점(직접 겪은 버그):
  - 이모지(📄 등)를 배너 텍스트에 넣으면 LibreOffice 렌더링에서 한글이 깨질 수 있다 → 이모지 쓰지 않는다.
  - `python-docx`의 "List Number" 스타일은 문서 전체에서 번호가 이어진다(섹션마다 1부터 다시 시작 안 됨) → 자동 스타일 대신 직접 "1. ", "2. " 텍스트를 붙인다.
  - 이미 저장된(한 번 저장 후 다시 연) docx에 `doc.add_page_break()`로 빈 문단을 만들고 바로 이어서 표(배너)를 추가하면, LibreOffice가 그 사이에 빈 페이지를 하나 더 끼워 넣는 버그가 있다 → `doc.add_page_break()` 대신, 배너 표 셀의 첫 문단에 `w:pageBreakBefore` 속성을 직접 넣는 방식(`pPr.insert(0, OxmlElement("w:pageBreakBefore"))`)을 쓰면 해결된다. (새로 처음부터 만드는 문서에서는 이 버그가 나타나지 않았음 — 기존 파일에 나중에 페이지를 추가할 때만 주의.)
- 만든 뒤에는 반드시 `soffice --headless --convert-to pdf` → `pdftoppm`으로 페이지 이미지 추출 → Read로 직접 눈으로 확인(한글 깨짐, 표, 체크리스트 정상 여부)하고 나서 최종본으로 확정한다.

## 3. 저장 위치 (두 곳 모두 동일하게)

새 프로젝트가 완성되면 아래 두 폴더 모두에 동일한 구조로 저장한다.

**`rungs-repo/워크시트/`** (docx와 pdf가 같은 폴더에 나란히 있음)
- `{프로젝트명}_엔트리_워크시트.docx`
- `{프로젝트명}_엔트리_워크시트.pdf`
- `Entry files/{프로젝트명}.ent`

**`Entry AI contents Project/워크시트/`** (rungs-repo 상위 폴더, pdf는 별도 `PDF/` 하위폴더에 저장하는 구조)
- `{프로젝트명}_엔트리_워크시트.docx`
- `PDF/{프로젝트명}_엔트리_워크시트.pdf`
- `Entry files/{프로젝트명}.ent`

파일명 규칙은 기존 파일들과 동일하게: 프로젝트명 그대로(한글/영문 혼용 가능) + `_엔트리_워크시트`.
